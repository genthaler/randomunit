 25   
  * great deal of cases automatically. Still, it is sensible to write specific
 26   
  * (usual) tests for rare corner cases, to complement the randomized search.
 27   
  * <p>
 28   
  * To implement a randomized test, we need some collaborating objects. These
 29   
  * objects are grouped into named <i>pools</i>, that denote objects with similar
 30   
  * semantics. For instance, an object pool named "keys" could be containing
 31   
  * objects that are used as keys in some map. By effectively organizing the
 32   
  * objects into groups, the search space is very much smaller, so the test
 33   
  * concentrate more on cases that actually matter.
 34   
  * <p>
 35   
  * The creation of a randomized test involves defining a subclass of this class,
 36   
  * and defining object pools and random test methods. Both these goals are
 37   
  * reached by simply annotating the test methods. An example follows, that
 38   
  * depicts a method that tests a special stack implementation that does not
 39   
  * accept null values.<BR>
 40   
  * <BR>
 41   
  * <p>
 42   
  *
 43   
  * <pre>
 44   
  * &#064;Prob(0.5)
 45   
  * &#064;Params({ &quot;stackPool&quot;, &quot;elementPool&quot; })
 46   
  * public void check(MyStack stack, Object element) throws Exception {
 47   
  *         precondition(element != null, &quot;Null elements are not allowed&quot;);
 48   
  *         precondition(!stack.isFull(), &quot;Stack cannot be empty&quot;);
 49   
  *

  *         int size = stack.size();

  *         stack.push(element);
 52   
  *

  *         postcondition(!stack.isEmpty(), &quot;Stack is not empty&quot;);

  *         postcondition(stack.size() == size + 1, &quot;Stack size grows by one&quot;);

  * </pre>
 57   
  * <p>
 58   
  * The &#064;Params annotation specifies the names of the respective object

  * pools, from which to choose random members to inject in the parameters of the

  * method. This concept is a form of dependency-injection: the method describes

  * its dependencies, and the "container" (the randomized test) injects

  * appropriate values. Note that &#064;Params is necessary as the type of the
 63   
  * arguments cannot express possible semantics differences between objects of
 64   
  * the same type (ie, objects of same class that are used in a substantially
 65   
  * different way). Please note that the methods are not forced to specify the
 66   
  * exact type of the arguments: any super type will do. Also, primitive
 67   
  * parameters can be defined. Yet, if a primitive argument is prescribed, and
 68   
  * the specified object pool contains null values, there is the possibility that
 69   
  * <code>null</code> will be randomly selected and attempted to be injected as a
 70   
  * primitive, but this is impossible and an exception will be thrown.
 71   
  * <p>
 72   
  * A random test method usually is composed by three parts: preconditions,
 73   
  * actual testing of some object via its methods, and postconditions. If a
 74   
  * precondition fails, nothing happens; this just means that the random
 75   
  * arguments that were chosen were illegal for the respective action to take

  * place. This is not counted as a complete step. But if the testing of the
 77   
  * object throws an exception, this constitutes a bug: valid arguments
 78   
  * (preconditions passed) caused a failed execution. If any postcondition fails,
 79   
  * this is also a bug: the object under test did not maintain its contract.
 80   
  * <p>

  * test method), can return a value and request it to be added to a object pool

  *                                                                 // Integer automatically

  * <p>

  * from (or anyhow depended on) other objects into consideration.

  * <p>

  * will be considered by JUnit as usual. If wanted, some prefix like "random"

  * can be used for method names, but this is not required.

  * <p>

  * When a random method is chosen, that needs objects from pools that are

  * <p>

  * <p>
